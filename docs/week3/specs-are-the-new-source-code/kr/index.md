# 명세서의 죽음, 명세서의 부활!

[원본 링크](https://blog.ravi-mehta.com/p/specs-are-the-new-source-code)

**저자:** Ravi Mehta, Danny Martinez
**게시일:** 2025년 7월 31일

## 서론

필자는 커리어 전반에 걸쳐 명세서(spec)가 점점 짧아지는 것을 목격해왔습니다. Microsoft에서는 PM의 가치를 명세서 무게로 측정했고, Tripadvisor에서 전설적인 명세서는 냅킨에 휘갈겨 쓴 메모였습니다. 하지만 팀들은 명세서를 가치 있는 작업이 아닌 필수적인 서류 절차 정도로 취급합니다. 엔지니어의 코드와 디자이너의 인터페이스는 칭찬하면서, 명세서 품질은 간과합니다.

이것은 근본적인 실수입니다. Ravi Mehta의 Product Competency Toolkit에서 기능 명세서(Feature Specification)는 12가지 역량 중 첫 번째이며, 실행력, 전달, 품질의 기반입니다. 훌륭한 명세서야말로 효과적인 프로덕트 매니지먼트의 기초입니다.

## 변화하는 환경

개발의 역학이 바뀌고 있습니다. 엔지니어들은 그 어느 때보다 빠르게 움직입니다. AI는 대략적인 개념을 몇 분 만에 작동하는 코드로 변환합니다. 병목이 구현에서 이해로 이동했습니다: **무엇을 만들어야 하는지 알고, 팀을 요구사항에 맞춰 정렬하는 것이 핵심입니다.**

한때 일시적인 서류 작업에 불과했던 명세서가 이제 프로덕트 매니지먼트의 기반이 되었습니다. 본질적으로 명세서 자체가 소스 코드가 된 것입니다.

## PM이 갑자기 병목이 된 이유

앤드류 응(Andrew Ng)은 최근 전례 없는 트렌드를 언급했습니다: "제 인생에서 처음으로 매니저들이 엔지니어의 두 배나 되는 PM을 두자고 제안하는 것을 봤습니다."

이것은 이전 예측을 확인해줍니다. 엔지니어가 AI로 가속화되면서 기업에는 PM이 더 적게 아니라 더 많이 필요합니다. 제품 가속화가 모든 PM 역량에 압력을 가중합니다. 고객 이해, 기능 설계, 임팩트 검증. 이 모든 수요가 하나의 산출물에 집중됩니다: 명세서입니다.

## 명세서가 새로운 소스 코드인 이유

전통적인 프로그래밍에서 사람이 읽을 수 있는 소스 코드는 최적화된 기계 코드로 컴파일됩니다. 소스 코드는 진정한 원천 자료(source of truth)로서 주석, 구조, 문서를 포함합니다.

OpenAI의 션 그로브(Sean Grove)는 도발적인 논제를 제시합니다: 잘 작성된 프롬프트(명세서)가 새로운 소스 코드를 구성한다는 것입니다. 현재 AI 개발은 이 논리를 뒤집습니다. 팀이 정성스럽게 프롬프트를 작성하고, AI가 코드를 생성하면, 팀은 프롬프트를 버리고 코드만 보존합니다.

그로브는 이렇게 관찰합니다: "이건 마치 소스를 파쇄하고 바이너리만 아주 신중하게 버전 관리하는 것과 같습니다."

코드는 명세서의 "손실 투영(lossy projection)"입니다. 코드를 읽어도 완전한 의도를 알 수 없습니다. 바이너리를 디컴파일해도 원래 주석이나 변수 이름을 복구할 수 없듯이 말입니다. 명세서는 모든 것을 담고 있습니다. 충분히 상세한 명세서가 있으면 "좋은 TypeScript, 좋은 Rust, 서버, 클라이언트, 문서, 튜토리얼, 블로그 포스트, 심지어 팟캐스트까지" 생성할 수 있습니다.

결정적으로, 명세서는 사람과 기계 모두를 공동의 목표에 정렬시킵니다. 그로브는 말합니다: "문서화된 명세서는 사람들을 효과적으로 정렬하고, 소통하고 토론하고 참조하고 동기화하는 데 사용하는 산출물입니다."

그는 예측합니다: "가까운 미래에, 가장 효과적으로 소통하는 사람이 가장 가치 있는 프로그래머가 될 것입니다. 문자 그대로, 효과적으로 소통할 수 있다면 프로그래밍할 수 있습니다."

희소한 기술은 코딩이 아닙니다. 완전한 의도와 가치를 담은 명세서를 작성하는 것입니다. PM에게 이것은 전통적인 업무입니다. 이제 기계가 듣고 있을 뿐입니다.

## 프로토타입이 명세서를 대체하지 못한 이유

역사적으로 제품 수명주기는 명세서가 와이어프레이밍, 디자인, 프로토타이핑, MVP 개발을 주도하는 방식으로 시작했습니다. 필요하지만 부담스러운 과정이었습니다.

이 패러다임이 근본적으로 바뀌었습니다: **이제 명세서는 종종 입력이 아니라 출력입니다.**

v0, Lovable, Replit 같은 현대적 도구들은 엔지니어링 없이도 몇 시간 안에 작동하는 프로토타입을 만들게 해줍니다. 단순히 더 빠른 것이 아니라 근본적으로 다릅니다. "바이브 코딩"된 프로토타입으로 기능 명세서를 작성하기 전에 실제 고객 피드백을 수집할 수 있습니다. 팀은 가정을 테스트하고, 플로우를 반복하고, 인터랙션을 다듬습니다.

**기존 워크플로우:** 모호한 아이디어 → 와이어프레임 → 디자인 → 엔지니어가 만든 MVP → 고객 피드백 → 고통스러운 명세서 수정 → 리디자인 → 다시 구현 → 불확실성.

**새로운 워크플로우:** 모호한 아이디어 → 빠른 프로토타입 → 고객 피드백 → 명확한 명세서 → AI 지원 구현.

프로토타입이 명세서를 없앤 것이 아닙니다. 명세서를 더 좋게 만들었습니다.

## 명세서 주도 개발 실전

decimals 창업자 대니 마르티네즈(Danny Martinez)가 실용적인 명세서 주도 개발을 보여줍니다. 그의 플랫폼은 크리에이터 이코노미 전문가들이 네트워크의 인재를 채용 기회에 연결하도록 돕습니다.

랜딩 페이지 출시 후, 팀은 전문가들에게 자신의 페이지 링크에 빠르게 접근할 방법이 필요했습니다. 마르티네즈의 공동 창업자가 메시지를 보냈습니다: "헤더에 company-apply 페이지로 이동하는 버튼이 필요해요. 저는 지금 이메일에 집중하고 있어서요. 이건 바이브 코딩으로 할 수 있으니까 맡아주세요."

**사용한 도구:**
- 프로젝트 관리: Linear
- IDE: VS Code
- 확장 프로그램: GitHub Copilot Pro
- 모델: Claude Sonnet 4
- MCP 서버: Linear MCP server

**프로세스 단계:**
1. Slack 메시지에서 Linear 티켓 생성
2. 티켓에 원하는 카피 명확히 작성
3. Copilot을 열고 Claude가 Linear 티켓에 접근하도록 프롬프트
4. Claude가 티켓을 코드베이스와 대조해 검토하도록 프롬프트
5. Claude가 브랜치를 만들고 변경사항을 구현하도록 프롬프트
6. 변경사항 테스트
7. GitHub PR 열기
8. 엔지니어 리뷰/승인 대기

핵심 인사이트: 비기술자가 전략적인 Claude 프롬프트만으로 Linear 티켓, 코드베이스, 엔지니어링 협업을 성공적으로 탐색했습니다.

**필수 요건:**
1. **구체적으로 작성하기:** 모호한 명세서는 지저분한 코드베이스를 만듭니다. Claude는 초기 티켓 초안을 다듬고 명세서 가이드라인을 수립하는 데 도움을 줍니다.
2. **선별하기:** 이 접근법은 단순한 작업에 적합합니다. 복잡한 티켓은 경험 있는 인력이 필요합니다.
3. **게이트키핑:** 엔지니어 리뷰가 단순함과 기능성 사이의 균형을 보장합니다.

규칙이 바뀌었습니다. 명세서는 LLM을 포함해 제품을 만드는 모든 사람의 원천 자료(source of truth)입니다.

적절한 설정이 있으면 비기술 기여자도 현실적으로 코드베이스에 기여할 수 있습니다. Claude에 계속 의존하는 대신 시스템을 점진적으로 이해하고 독립적으로 변경사항을 구현하게 됩니다.

잠재적으로 AI 에이전트가 창작자가 커피 마시는 동안 티켓을 완료할 수도 있습니다.

## 결론

윌리엄 깁슨(William Gibson)은 이렇게 말했습니다: "미래는 이미 여기에 있다. 다만 고르게 분포되어 있지 않을 뿐이다."

AI의 이득은 여전히 극심하게 불균등합니다. 코드, 텍스트, 이미지 생성은 양자 도약을 이루며 "AI 속도"로 작동합니다. 고객 대화, 니즈 발굴, 구매 설득은 여전히 "인간 속도"입니다.

이러한 불균등한 분포가 제품 팀을 재편하고 있습니다. 초점이 구현에서 이해로 이동합니다. 전통적인 PM 스킬인 고객 니즈 이해, 명확한 문제 정의, 우아한 솔루션 설계가 기하급수적으로 가치가 높아졌습니다.

뛰어난 PM은 인사이트를 명세서로 변환합니다. 팀을 정렬하고, 구현을 가이드하고, 점점 자동화되는 개발 환경에서 지속적인 참조 자료가 되는 산출물로 말입니다.

## 핵심 요약

- 명세서가 일시적인 서류 작업에서 제품 개발의 기반으로 변화함
- AI가 엔지니어를 가속화하면서 PM의 역할과 명세서 작성 능력이 더 중요해짐
- 션 그로브: 명세서는 새로운 소스 코드, 코드는 명세서의 "손실 투영"
- 바이브 코딩 도구로 프로토타입 먼저, 명세서 나중 워크플로우가 가능해짐
- 비기술자도 잘 작성된 명세서와 AI 도구로 코드베이스에 기여 가능
- 미래의 가장 가치 있는 프로그래머는 가장 효과적으로 소통하는 사람
