# 코딩 에이전트 101: 일을 실제로 완수하는 기술

[원본 링크](https://devin.ai/agents101)

## 소개

이 가이드는 "특정 제품에 종속되지 않는", "실용적", "기술적" 성격을 표방합니다. 2025년에 코딩 에이전트와 효과적으로 협업하려는 엔지니어를 위해 작성했습니다. 고객 경험과 Cognition 팀이 Devin을 개발하며 얻은 교훈을 담았습니다.

## 시작하기: 프롬프트 기초

### 무엇을 원하는지보다 _어떻게_ 해야 하는지 알려주세요

일반적인 요청 대신 상세한 방법론을 제공하세요. 가이드에서 설명하듯이, 에이전트를 "의사결정이 불안정한 주니어 코딩 파트너"로 대하세요.

**예시:** 단위 테스트를 일반적으로 요청하지 말고, 테스트할 기능을 지정하고, 엣지 케이스를 식별하고, 모킹 요구사항을 명확히 하세요.

### 에이전트에게 시작점을 알려주세요

저장소, 관련 문서, 핵심 컴포넌트를 안내하세요. 이렇게 하면 불필요한 시행착오를 줄일 수 있습니다.

**예시:** "[여기] 최신 문서를 참고해서 model groups 디렉토리에 새 구현 파일을 만들어줘."

### 방어적 프롬프팅을 실천하세요

신입 인턴에게 하듯이 혼란스러울 수 있는 부분을 미리 예상하고 선제적으로 지침을 명확히 하세요.

**예시:** "주의해야 해. 테스트 전에 코드를 변경할 때마다 바인딩을 다시 컴파일해야 해."

### CI, 테스트, 타입, 린터에 접근 권한을 부여하세요

테스팅 도구와 타입 체커를 통한 강력한 피드백 루프가 에이전트 성능을 향상시킵니다. 가이드에 따르면 팀은 "대부분 타입이 없던 Python SDK에서 완전히 타입이 적용된 SDK로" 전환했습니다.

### 당신의 전문성을 활용하세요

인간의 감독은 여전히 필수입니다. "궁극적으로, 코드의 최종 정확성은 당신이 책임집니다."

## 워크플로우에서 에이전트 활용하기

### 새 작업을 즉시 처리하세요

빠른 조사나 변경 작업을 위임해서 주요 업무에 집중하세요. 많은 팀이 "버그 수정이나 사소한 기능 업데이트를 논의할 때 Slack에서 @Devin을 태그합니다."

### 이동 중에도 코딩하세요

모바일로 접근할 수 있어서 여행이나 출퇴근 중에도 긴급한 이슈를 처리할 수 있습니다.

### 잡무를 넘기세요

커밋 이분법(bisecting)이나 문서 업데이트 같은 반복 작업을 위임하세요.

### 분석 마비를 피하세요

에이전트에게 여러 아키텍처 접근 방식을 구현하게 해서 구체적으로 비교하세요.

### 프리뷰 배포를 설정하세요

각 PR마다 자동으로 프리뷰 배포가 생성되도록 CI/CD 파이프라인을 구성하세요.

## 중급: 더 큰 티켓 위임하기

### 첫 번째 초안을 자동화하세요

큰 작업의 경우, 에이전트가 초기 PR 초안을 만들 수 있습니다. 가이드에서는 이렇게 비유합니다: "자율 에이전트는 초기 계획을 기반으로 작업을 시작하고 첫 번째 초안 솔루션을 만듭니다."

현실적 기대치는 "완전한 자동화가 아닌 약 80% 시간 절약"입니다.

### PRD(제품 요구사항 정의서)를 공동 개발하세요

복잡하거나 모호하게 정의된 작업은 에이전트와 함께 상세한 계획을 수립하세요. 에이전트는 "우리 인증 시스템은 어떻게 작동하나요?"와 같은 탐색 질문을 조사할 수 있습니다.

### 체크포인트를 설정하세요

여러 부분으로 나뉜 작업은 명확한 중단점을 설정하세요: "계획 → 청크 구현 → 테스트 → 수정 → 체크포인트 리뷰 → 다음 청크"

### 에이전트가 스스로 검증하도록 가르치세요

테스팅 프로세스를 명시해서 에이전트가 향후 작업을 독립적으로 검증할 수 있게 하세요.

### AI 핫스팟의 테스트 커버리지를 높이세요

중요한 코드 마이그레이션이나 리팩터링을 위임하기 전에 자주 수정되는 영역의 테스트를 강화하세요.

## 고급: 워크플로우 자동화

### 반복 작업용 단축키 만들기

일반적인 자동화 후보로는 피처 플래그 제거, 의존성 업그레이드, 피처 PR에 대한 테스트 생성이 있습니다.

### 지능형 코드 리뷰 및 강제 적용

에이전트는 일부 경우 기존 린트 규칙보다 더 정확하게 일반적인 실수를 점검할 수 있습니다.

### 인시던트와 알림에 연결하기

에이전트는 API나 CLI 통합으로 자동 트리거될 수 있습니다. 그러나 가이드에서는 프로덕션 이슈에 대해 "AI의 디버깅 능력이 그리 뛰어나지 않다"고 주의합니다. 엔드투엔드 수정을 시도하기보다 의심스러운 오류를 플래그하세요.

## 커스터마이징 및 성능 향상

### 환경 설정

에이전트 환경을 팀 구성과 정확히 맞추세요. 언어 버전, 의존성, 사전 구성된 인증 등을 포함합니다.

### 커스텀 CLI 도구와 MCP 빌드

Linear 티켓 정보 가져오기나 개발 환경 재시작 같은 반복 워크플로우용 스크립트를 만드세요.

### 에이전트의 지식 베이스에 추가하기

프레임워크별 패턴, 프로젝트 아키텍처, 테스팅 규칙, 권장 도구에 대한 피드백을 문서화하세요.

## 실용적 고려사항

### 자율 에이전트의 한계

**제한적인 디버깅 능력:** 완전한 수정을 기대하지 말고 가능한 근본 원인을 요청하세요.

**세밀한 시각적 추론 부족:** 픽셀 단위로 완벽한 Figma 목업 매칭을 기대하지 말고 재사용 가능한 컴포넌트가 있는 디자인 시스템을 사용하세요.

**지식 단절:** 에이전트는 사전 학습된 지식에 의존합니다. 새 라이브러리는 "명시적으로 최신 문서를 가리켜 주세요."

### 시간 관리 및 손실 최소화

**손실을 빨리 끊으세요:** 근본적인 오해 징후가 보이면 상호작용을 중단하세요.

**실험을 다양화하세요:** 다양한 프롬프트를 시도하고 성공적인 작업 카테고리에 집중하세요.

**새로 시작하세요:** 막혔을 때 "에이전트는 사람보다 훨씬 더 자주 처음부터 다시 시작하는 것이 정답입니다."

### 보안 및 권한

- 일회용 이메일로 전용 에이전트 계정 생성
- 개발/스테이징 환경만 제공
- 가능하면 읽기 전용 API 접근 권한 부여
- 외부 서비스와 상호작용하는 스크립트는 사람이 수동으로 실행

## 결론

가이드에서는 역량이 발전해도 "소프트웨어 엔지니어는 사라지지 않을 것"이라고 강조합니다. 자동화가 개인의 영향력을 증폭시키고 병렬 작업 관리를 가능하게 해도 "프로젝트, 시스템, 코드에 대한 진정한 소유권은 그 어느 때보다 중요합니다."

## 핵심 요약

- 코딩 에이전트에게 무엇을 할지보다 **어떻게** 할지를 상세히 알려주는 것이 중요합니다.
- 에이전트를 "주니어 코딩 파트너"로 대하고, 방어적 프롬프팅을 실천하세요.
- CI, 테스트, 타입 시스템을 통한 강력한 피드백 루프가 에이전트 성능을 향상시킵니다.
- 현실적 기대치: 완전 자동화가 아닌 약 80% 시간 절약.
- 복잡한 작업은 체크포인트를 설정하고 단계별로 진행하세요.
- 에이전트의 한계(디버깅 능력, 시각적 추론, 지식 단절)를 인지하세요.
- 막혔을 때는 사람보다 에이전트에서 더 자주 "처음부터 다시 시작"이 정답입니다.
- 보안을 위해 전용 계정, 제한된 환경, 읽기 전용 권한을 사용하세요.
